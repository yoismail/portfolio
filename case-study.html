PayFlow Database Optimization II

This project is where my portfolio stops sounding generic and starts showing how I really think. I built it to demonstrate a rerunnable ETL workflow with clear boundaries between transformation, warehouse loading, and reset logic.

Repo-backed facts

  • The repository explicitly separates raw → processed transformation from processed → analytics warehouse loading.
  • It includes wipe/reset commands so stale files and schemas can be cleared before reruns.
  • The analytics stage builds validated dimensions and fact tables inside PostgreSQL.
  • A runner script executes the full flow end to end rather than relying on manual steps.
Project story

Why I built this the way I did

The key idea was simple: I wanted the repo structure itself to demonstrate engineering judgement, not just the final output tables.

The problem I wanted to solve

Raw commerce data is rarely analysis-ready. In this case, customers, sellers, orders, items, and payments exist as separate sources. I did not want to jump straight to charts or SQL queries. I wanted to show the intermediate engineering work required to clean, align, validate, and reload that data properly.

The portfolio problem I wanted to solve

A lot of early-career portfolios say “ETL pipeline” but still behave like one-off scripts. I built PayFlow so a recruiter could see the difference between a data task and a delivery pattern: staged transformation, schema-aware loading, reset controls, and one-command orchestration.

Delivery pattern

I separated the pipeline on purpose.

That decision is the backbone of this project and the main reason it works so well as a flagship case study.

1. Raw to processed

I used the transformation layer to normalise columns, rename fields to match analytics expectations, clean entity tables, convert timestamps, and save curated datasets into a processed data folder. This stage is about making source data dependable before analytics loading begins.

2. Processed to analytics

Once the processed datasets are stable, the warehouse loader validates schema, builds dimensions, constructs fact-style tables, and loads them into PostgreSQL. That keeps warehouse logic separate from source cleanup and makes debugging less messy.

3. Reset and rerun

I added wipe/reset tooling because rerun safety is part of maintainability. Being able to clear raw, processed, analytics, or everything at once makes the project more practical during iteration and closer to how disciplined engineering workflows behave.

Architecture

The value is in the boundaries.

I wanted each layer to have a clear responsibility so the project stayed understandable as it grew.

Transform layer

This stage handles field alignment, column normalisation, timestamp conversion, and dataset cleanup. It produces processed outputs that are structurally ready for analytics logic.

Warehouse layer

This stage is where dimensions and facts are built and loaded into the analytics schema. Keeping it separate makes warehouse design a visible part of the implementation, not an afterthought.

Operational discipline

Reset scripts, structured folders, and a single run command make the project easier to explain and easier to rerun. That operational discipline is part of the story I want employers to see.
