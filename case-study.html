Problem → Solution → Architecture

The problem was not only loading data. It was preventing the workflow from becoming fragile as the logic evolved.

I wanted the project to be easier to change without losing control of the output. That meant separating responsibilities, keeping warehouse logic explicit, validating data quality at multiple stages, and making full reruns predictable.

Problem

When extraction, cleaning, table creation, and analytics loading are mixed together, every schema change increases debugging friction. The workflow becomes harder to trust, and reruns become riskier.

RiskTangled responsibilities
ImpactUnclear reruns and slower debugging

Solution

I separated the workflow into raw ingestion, staging clean-up, transform logic, analytics loading, and orchestration. That made failures easier to trace and made warehouse behavior more deliberate.

MethodStage separation
BenefitClearer ownership and maintainability

Architecture Choice

I treated each layer as a separate engineering concern: raw data as source truth, staging for clean standardized tables, transform for dimensional preparation, analytics for warehouse delivery, and orchestration for repeatable runs.

Delivery Process

How the pipeline is structured from reset to warehouse delivery.

The repo makes the process explicit: reset the environment, extract the source data, explore and validate it, clean and stage it, then transform and load a star-schema warehouse through orchestrated execution.

01

Wipe Phase

Resets schemas and data folders so the environment returns to a clean state before execution. This supports deterministic reruns during redesign and debugging.

02

Extract + Explore

Downloads and validates the raw CSV data, then logs shapes, heads, dtypes, row counts, and missing values to make the input easier to reason about.

03

Clean + Stage

Cleans customers, sellers, and transactions, handles cancellations and timestamps, writes cleaned outputs, and loads standardized tables into staging.

04

Transform + Load

Builds dimensions and fact tables, maps keys, preserves referential integrity, and loads the analytics schema through an orchestrated, logged workflow.

This pipeline enables cleaner reporting and analytics by moving from raw operational data into structures designed for BI-style queries, repeatable execution, and easier future extension.
