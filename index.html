Why this project matters

I built PayFlow around the Brazilian Olist dataset to show I can take fragmented source tables and turn them into a workflow that is easier to rerun, easier to reason about, and easier to extend. Instead of doing everything in one script, I split the work into transformation, warehouse load, reset, and orchestration stages.

That separation is deliberate. It reflects how I think about delivery in real environments: you need a reliable way to clean source data, a controlled way to load analytics models, and a safe way to rerun the pipeline when something changes.

Delivery pattern I used

  • Raw source files are cleaned and standardised into processed datasets before warehouse loading begins.
  • Processed data is then validated and loaded into dimension and fact-style structures inside PostgreSQL.
  • Reset tooling clears stale layers so reruns behave predictably during iteration.
  • A runner script ties the workflow together so the repo behaves like a system rather than a folder of disconnected scripts.
